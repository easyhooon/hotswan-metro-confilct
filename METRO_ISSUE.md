# Generated graph callsite can become stale after @AssistedInject ViewModel constructor signature changes

Hi, I prepared a minimal repro project for an assisted ViewModel factory signature mismatch:

https://github.com/easyhooon/hotswan-metro-confilct

## Environment

- Kotlin: 2.4.0
- AGP: 9.0.1
- Compose BOM: 2026.06.00
- Metro / metrox-viewmodel / metrox-viewmodel-compose: 1.2.1
- HotSwan compiler plugin: 2.0.0-beta03
- Android Studio Live Edit: disabled

## Project shape

The repro project has:

- `:app` module containing the Metro `AppGraph` / `ActivityRetainedGraph`
- `:feature` module containing:
  - `SampleScreen`
  - `@AssistedInject SampleViewModel`
- `SampleViewModel` is contributed with `@ContributesIntoMap(ActivityRetainedScope::class)`
- `SampleViewModel` is created from Compose via `assistedMetroViewModel(...)`

The current baseline constructor has one assisted argument and two non-assisted injected dependencies:

```kotlin
@AssistedInject
class SampleViewModel(
    @param:Assisted private val screenName: String,
    private val firstRepository: FirstRepository,
    private val secondRepository: SecondRepository,
) : ViewModel()
```

## Reproduction steps

1. Clone the repro project:

```bash
git clone https://github.com/easyhooon/hotswan-metro-confilct
```

2. Open it in Android Studio.

3. Make sure Android Studio Live Edit is disabled.

4. Build and install the app once with the current source.

5. Run the app and confirm the screen renders.

6. In `SampleViewModel`, remove the non-assisted injected dependency:

```kotlin
import com.easyhooon.hotswanmetroconflict.data.SecondRepository
```

and:

```kotlin
private val secondRepository: SecondRepository,
```

7. Change the message to only use `FirstRepository`:

```kotlin
val message: String = "$screenName ${firstRepository.label()}"
```

8. Apply/deploy the change through Android Studio / HotSwan.

## Actual behavior

The generated assisted ViewModel factory signature changes, but the generated Metro graph callsite can remain stale.

In my real project, this produced a mismatch like:

```text
TripInviteNameSelectViewModel$MetroFactory$Companion exposes:
create(Provider<InvitationRepository>)
```

but:

```text
AppGraph$Impl$ActivityRetainedGraphImpl still invokes:
create(Provider<InvitationRepository>, Provider<TripRepository>)
```

This results in:

```text
java.lang.NoSuchMethodError:
No virtual method create(Ldev/zacsweers/metro/Provider;Ldev/zacsweers/metro/Provider;)...
```

The stack points into the generated graph:

```text
at ...AppGraph$Impl$ActivityRetainedGraphImpl.getMapOfKClass_out_ManualViewModelAssistedFactory_ToManualViewModelAssistedFactoryProvider(...)
at ...AppGraph$Impl$ActivityRetainedGraphImpl.<init>(...)
```

## Expected behavior

When an `@AssistedInject` constructor signature changes and Metro regenerates the assisted factory signature, the generated graph callsite should also be invalidated/regenerated.

The build should not produce bytecode where:

- `SampleViewModel$MetroFactory$Companion` exposes the new `create(...)` descriptor
- `AppGraph$Impl$ActivityRetainedGraphImpl` invokes the old/stale `create(...)` descriptor

## Bytecode inspection

The repro README includes commands for checking both sides:

```bash
javap -classpath feature/build/intermediates/built_in_kotlinc/debug/compileDebugKotlin/classes \
  -s -p 'com.easyhooon.hotswanmetroconflict.ui.SampleViewModel$MetroFactory$Companion' \
  | rg -n "create|Provider"
```

```bash
javap -classpath app/build/intermediates/built_in_kotlinc/debug/compileDebugKotlin/classes:feature/build/intermediates/built_in_kotlinc/debug/compileDebugKotlin/classes \
  -c -p 'com.easyhooon.hotswanmetroconflict.di.AppGraph$Impl$ActivityRetainedGraphImpl' \
  | rg -n "SampleViewModel|MetroFactory\\$Companion.create" -C 4
```

## Notes

This originally surfaced while testing HotSwan because the stale generated graph callsite was executed during Compose composition, causing a blank white screen after Compose captured the `NoSuchMethodError`.

However, after inspecting the generated bytecode, the mismatch appears to be between Metro-generated classes themselves, so this looks related to Metro generated graph invalidation / incremental compilation / build cache behavior.
